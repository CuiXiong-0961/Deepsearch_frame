# 2026-04-01：LangGraph 并行化 + Redis 结果外置 + 日志策略改造记录

本文记录一次针对 `Deepsearch_frame` 的架构升级需求，目标是将当前串行执行的 Deep Research 主流程迁移到 **LangGraph**，支持 **按子任务并行**、**按需合并日志**、并通过 **Redis 外置存储检索/中间结果**，避免图状态（state）过大导致内存压力。  
（本文件为需求与设计记录，不包含实现代码；后续可继续补充。）

---

## 背景与问题

当前主流程位于 `orchestrator/runner.py`，按 `plan.subtasks` **线性串行**执行：

- `Planner.create_plan(task)`
- 对每个 subtask 执行：`web_search → reflection →（可选补搜）→ summarize_subtask`
- 最后执行：`generate_outline → generate_report`

随着子任务数量增加、每步调用 LLM/检索耗时增加，**串行执行过慢**。同时，如果将所有检索结果/页面正文直接塞进图的 state，容易造成 **state 体积过大、内存膨胀甚至 OOM**。

---

## 总目标

- **LangGraph 重写编排**：用图来表达 Planner → 子任务并行 → 汇总合成。
- **并行执行子任务**：对子任务之间互不影响的部分并行，加速整体完成时间。
- **日志可回溯、可合并**：每个子任务单独记录步骤日志，运行结束后按 subtask 顺序合并为一个总日志文件。
- **Redis 外置存储中间结果**：检索/文档等大对象不放入 state；state 仅保存 Redis key。
- **延迟清理 Redis**：一轮问题的 Redis 记录不立即删除，等“用户问下一问问题”时或者点击评论“好或者坏”后再清理上一轮记录。

---

## 并行化策略（核心设计）

### 并行粒度

- **并行单元**：每个 `SubTask` 的处理链路（检索→反思→补搜→总结）。
- **保持串行**：
  - `Planner.create_plan`：必须先得到 subtasks 才能 map。
  - 汇总合成（outline/report）：需要收集所有 subtasks 的总结结果。

### 并行条件

仅当子任务间 **不会相互影响、不依赖彼此产生的中间结论/文档池** 时，才允许并行。  
（默认假设：各 subtask 独立；如未来引入跨 subtask 的共享记忆或全局去重/学习，应重新评估并行边界。）

### 并发控制

新增 CLI 参数：

- `--parallel N`：并行度上限（默认 **3**）

并发控制必须是**全局**的（同时限制 Tavily / page fetch / LLM 调用），避免：
- API 429（限流）激增
- 连接池/线程池/CPU 飙升
- 日志写入竞争

## LangGraph 图结构（建议形态）

建议将整体拆为三段：

1. **Plan 节点**
   - 输入：`task`
   - 输出：`subtasks[]`（仅小对象）

2. **Subtask Worker（并行 map）**
   - 对每个 subtask 独立运行：
     - search / enrich / compress / reflect / extra rounds / summarize
   - 输出：`SubtaskResult`（轻量化）
     - `subtask_id`
     - `subtask_content`
     - `summary`
     - `redis_keys`（指向 docs、audit、页面正文、reflection 等大对象）
     - `log_path`（该子任务日志文件路径）

3. **Join / Synthesis 节点**
   - 输入：按 subtask 顺序排列的 `SubtaskResult[]`
   - 输出：`outline`、`report`
   - 可将最终 `report` 返回 UI/CLI

> 关键原则：**state 中不存文档全文，不存大列表**；仅保存 key、路径、摘要等小对象。

## Redis 外置存储设计

### 设计目标

- 检索结果、页面正文、文档池、反思输出等“大对象”写入 Redis；
- LangGraph state 只保留 Redis key（或 key 列表）；
- 降低 state 序列化/传递成本，避免爆内存。

### 存储内容（建议）

- 每轮用户问题（run/session）生成唯一 `run_id`（或 `conversation_id + turn_id`）。
- Redis key 命名建议带命名空间：
  - `deepsearch:{run_id}:subtask:{subtask_id}:docs`
  - `deepsearch:{run_id}:subtask:{subtask_id}:audit`
  - `deepsearch:{run_id}:subtask:{subtask_id}:reflection`
  - `deepsearch:{run_id}:subtask:{subtask_id}:summary_materials`
  - `deepsearch:{run_id}:meta`（记录本轮所有 key，便于统一清理）

### 清理策略（强约束）

- **不在本轮结束时清理**；
- **等用户下一问开始时清理上一轮 run_id 的所有 key**；
- 若存在并发对话/多用户场景，需以 “会话维度” 隔离（避免误删他人 run）。

## 日志策略改造

### 目标

- 每个子任务单独一个日志文件，避免并行写同一文件导致乱序/竞争；
- 运行结束后按 subtask 顺序合并，生成一个“总日志”文件，便于整体回溯。

### 具体要求

- **每个子任务一个日志**：
  - 文件名可包含：`run_id`、`subtask_id`、时间戳
- **合并日志**：
  - 合并顺序：按 plan 中 subtask 的顺序（而非实际完成顺序）
  - 合并输出写入 `logger/records/`（与现有体系兼容）
  - 合并时建议加分隔符与子任务 header，避免内容混淆

### 最终方案选择（已定）

选择 **“先写一系列小日志文件 → 运行结束后按 subtask 顺序拼接为总日志”** 的方案；**不把日志主体存进 Redis**。

- **原因**：
  - **文件日志更适合长文本**：txt 合并与人工排障简单直接；Redis 更适合结构化 key/value 与中间态索引。
  - **避免 Redis 占用被日志撑爆**：检索正文已经会占用较多空间，日志再入 Redis 容易造成容量与成本压力。
  - **并行写更安全**：每个 subtask 独占文件，不需要跨线程/进程的复杂锁与顺序一致性设计。
- **Redis 仍会参与“日志管理的元数据”**（可选）：
  - `deepsearch:{run_id}:meta` 里可记录：每个 subtask 的 `log_path` 列表与合并后的 `merged_log_path`，便于下一问清理时同步做“文件侧清理策略”（例如评分为好则删除等）。

## CLI / 配置项

- `--parallel N`：并行度上限，默认 `3`
- Redis 连接相关（建议用环境变量或 `.env`）：
  - `REDIS_URL` 或 `REDIS_HOST/REDIS_PORT/REDIS_DB`
  - 可选：`REDIS_TTL_SECONDS`（如果需要兜底过期，避免长期积压）

## 与 UI（Gradio）联动（后续扩展点）

- UI 可展示：
  - 总进度（最少步数估计 + 已完成子任务数）
  - 子任务列表（按 plan 顺序逐行展示）
    - 每行：`{subtask_id}. {subtask_content}`
    - 若已完成：在行尾追加 `（已完成）`
    - 若未完成：保持原样
  - 当前任务指示（单独一行）：`当前任务：{subtask_id}` 或 `当前任务：{subtask_content}`
  - 动态更新规则：
    - 当 planner 产出 subtasks 后：立即渲染“子任务列表（全量未完成）”
    - 每当某个 subtask 完成（summarize 完成即可视为完成）：更新该行 `（已完成）`
    - 当并行执行导致“同时跑多个 subtask”时：
      - UI 的 “当前任务” 字段展示 **最近一次上报的步骤所属 subtask**（或展示一个“正在执行：s1, s3, s4”列表，后续再扩展）
    - 若规划（plan）发生变化（例如重新规划/插入新 subtask）：
      - UI 必须刷新任务列表（新增/删除/内容变化都要同步更新）
- UI 的“下一问”触发点：
  - 进入下一问之前调用清理：删除上一轮 run_id 的 Redis key 集合

## 风险与约束（必须考虑）

- **限流与重试**：并行会把调用量“同时刻”抬高，需要明确的并发上限与退避策略。
- **日志一致性**：子任务日志 + 合并日志，要确保路径、编码、换行、截断策略一致。
- **Redis 成本/容量**：页面正文可能很大，需考虑压缩、截断、按需存储。
- **可观测性**：建议在 `run_id` 层面记录本轮 key 列表，便于一键清理与排障。
