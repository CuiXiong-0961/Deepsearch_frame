# planner

## 定位
负责把用户任务拆解为可执行子任务，并在执行过程中动态调整计划。

## 负责内容
- 将原始任务转换为结构化 Plan（子任务列表、优先级、状态）。
- 约束子任务粒度，避免过粗或过碎。
- 定义子任务完成标准，供 Reflection 判断是否证据充分。
- 根据执行反馈更新 Plan（补充子任务、调整优先级、重排顺序、关闭无效任务）。

## 总体实现思路
- 输入为任务描述与上下文摘要，输出为标准化 Plan 数据结构。
- 将计划更新设计为可追踪版本变更，确保“为什么修改计划”可解释。
- 与 Reflection、Memory 协作：当证据不足或方向偏移时触发计划修正。
- 使用明确的计划质量规则（覆盖度、依赖关系、优先级一致性）保证计划可执行。

## 边界约束
- 不直接执行检索、不直接产出最终报告。
- 只负责“做什么、先做什么、什么时候调整”。

### 提示词及fewshot

举个例子，用户输入"分析中国新能源汽车 2025 年的竞争格局"，规划器会拆解成：

- 子任务 1：当前市场份额分布（比亚迪/特斯拉/蔚来/理想/小米各占多少）
- 子任务 2：各厂商的技术路线对比（纯电/混动/增程）
- 子任务 3：政策环境变化（补贴退坡/碳积分/出口限制）
- 子任务 4：供应链瓶颈分析（电池/芯片/稀土）
- 子任务 5：综合分析与趋势预测

研究任务：{task}

3. 最后一个子任务应该是综合分析/结论
4. 标注每个子任务的优先级（P0必须/P1重要/P2补充）

输出格式：
[子任务1] (P0) 具体问题描述
[子任务2] (P0) 具体问题描述
..."""
