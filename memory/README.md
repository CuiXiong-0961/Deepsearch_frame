# memory

## 定位
作为 Memory Hub，统一承载全流程中间状态与关键产物，是模块解耦核心。

## 负责内容
- 存储任务执行中的关键对象：Plan、查询历史、文档集合、评估结果、子任务小结、最终报告素材。
- 提供按任务、子任务、轮次、来源等维度的查询与回溯能力。
- 管理短期运行态与可选长期记忆，支持跨轮复用与连续研究。
- 维护数据一致性与版本信息，保证状态可追踪。

## 总体实现思路
- 统一读写入口，避免各模块私有状态分散导致耦合。
- 将写入设计为事件化或版本化记录，支持问题定位与执行复盘。
- 对外暴露稳定数据契约，确保 Planner、Retriever、Reflection、Synthesizer 解耦协作。
- 对关键数据做分层管理：热数据用于当前流程，冷数据用于归档与长期检索。

## 边界约束
- 不承担流程编排，不替代业务决策。
- 只提供“状态统一存取与可追踪记忆”能力。

###  细节把控

搜索完所有子任务后，系统面前可能有几万字的素材。直接全部塞给 LLM 让它写报告？不行——即使上下文窗口够大，LLM 对超长输入的利用效率也会急剧下降（Lost in the Middle 问题）。

**方案：分层摘要 + 大纲驱动生成。**

**第一步：子任务级摘要。** 对每个子任务的搜索结果分别生成一段 500-800 字的摘要，保留关键数据和来源引用。这一步把几万字压缩到了几千字。

**第二步：大纲生成。** 基于所有子任务摘要，生成报告的大纲（一级标题+二级标题+每节要点）。

**第三步：分节生成。** 按照大纲，一节一节地生成正文。每次生成一节时，输入包括该节对应的子任务摘要 + 相邻节的摘要（保证上下文衔接）+ 大纲全文（保证全局一致性）。

**第四步：全文审校。** 最后对完整报告做一遍审校——检查逻辑一致性、数据矛盾、引用完整性。
